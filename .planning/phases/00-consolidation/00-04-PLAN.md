       ```markdown
       # ADR 0012 — Mirror protocol: SPS/PPS forwarding via silent JSON extension (Route A)

       **Status**: Accepted
       **Date**: 2026-04-20
       **Scope**: services/provider-node/internal/mirror/*

       ## Context

       Phase 0 / DEPLOY-03 closes the fan-out gap where a mirror-recipient
       provider cannot serve HLS init segments correctly because the origin
       never hands it the H.264 parameter sets (SPS/PPS) captured from the
       WHIP ingest. Without SPS/PPS, mirror-side HLSMuxer emits init
       segments referencing a PPS the decoder never saw, and viewers who
       join mid-session via the mirror path fail to decode the first IDR.

       Origin already captures SPS/PPS via `whip.Session.VideoSPSPPS()`
       (services/provider-node/internal/whip/whip.go:233) from either
       the WHIP SDP's `sprop-parameter-sets` fmtp line OR inline RTP
       STAP-A aggregates. On the origin path, `cmd/provider/main.go:443`
       feeds them to `whip.NewHLSMuxer(sess.ID, sps, pps)`. On the mirror
       path, `cmd/provider/main.go:280` passes `nil, nil` because the
       mirror stream protocol has no channel for parameter sets.

       Two routes to close this gap:

       - **Route A — silent JSON extension**: Add an optional
         `sps_pps_set: [{sps, pps}]` field to the existing
         `FrameTypeHeader` JSON body. Go's `json.Unmarshal` silently
         drops unknown keys on older readers, so a B4-emitting origin
         can populate the field without breaking pre-B4 mirrors.
       - **Route B — new FrameType**: Add `FrameTypeSPSPPS = 0x07` and
         emit a dedicated frame between the header and the first RTP.
         Requires softening `ReadAnyFrame`'s `default:` branch (which
         currently errors on unknown FrameTypes) so forward-compat is
         maintained. Also requires a deploy-ordering constraint: every
         mirror-side reader must run a B4-aware binary before any origin
         starts emitting 0x07 frames.

       ## Decision

       Adopt **Route A**: extend `Header` struct with
       `SPSPPSSet []SPSPPSPair` (JSON key `sps_pps_set`, `omitempty`).
       The origin MUST populate this field when
       `whip.Session.VideoSPSPPS()` returns non-nil bytes. A mirror-side
       reader MUST consult `header.SPSPPSSet` and, when non-empty, pass
       the first pair's SPS+PPS bytes to `whip.NewHLSMuxer` instead of
       `nil, nil`. When empty or nil, the mirror falls back to the
       existing inline STAP-A cache path in HLSMuxer.

       ## Consequences

       ### Positive
       - **Zero breaking change.** Pre-B4 mirrors running in production
         right now silently drop the new JSON key on ReadHeader. This is
         mechanically identical to how `CodecInfo.Channels` was added
         historically — additive, tested, safe.
       - **No deploy ordering.** A rolling deploy can swap any node at
         any time in any direction without protocol mismatch.
       - **No API break in `ReadAnyFrame`.** The `default:` branch stays
         strict (unknown FrameType = error), preserving the
         protocol-violation detection we rely on for malformed peers.
       - **Trivial to test.** Three unit tests in
         `internal/mirror/protocol_test.go` cover round-trip,
         legacy-read, and on-wire omission.

       ### Negative / Accepted trade-offs
       - JSON-in-Header is already the shape for codec metadata, so
         adding more keys there couples "stream shape" with "encoder
         hints" more tightly. We accept this because Route B has the
         same semantic coupling (it's still pre-RTP metadata, just
         split across frames), at the cost of wire-format churn we do
         not need.
       - Binary bytes serialised as JSON = base64-encoded on the wire
         (Go's default `[]byte` JSON behavior). Overhead is ~33% on the
         SPS+PPS payload, ~50 bytes total per session. Negligible.

       ## Rejected Alternatives

       ### Route B — new FrameTypeSPSPPS = 0x07
       Rejected because:
       - Forces softening `ReadAnyFrame`'s `default:` branch to `nil`
         return on unknown FrameType for forward-compat, weakening the
         protocol-violation detection.
       - Imposes a deploy-ordering constraint incompatible with the
         rolling-deploy posture locked in CONTEXT.md D-04.
       - Adds a new wire-format invariant (read frame 0 Header → read
         frame 1 SPS/PPS → read RTP frames) with no additional capability
         over Route A.
       - No evidence in the existing codebase of stream-level metadata
         that cannot fit in the JSON header; pion's `sprop-parameter-sets`
         and gohlslib's `codecs.H264.SPS/PPS` are both byte slices.

       ### Inline STAP-A capture only (no forwarding)
       Rejected because it requires the mirror to observe the first
       IDR before it can emit a valid init segment, so late-joining
       viewers face a variable ~1-5s race between mirror stream start
       and first IDR. Origin already has parameter sets out-of-band via
       SDP on most browsers (observed on Firefox + Safari); forwarding
       them removes the race entirely.

       ## References
       - `services/provider-node/internal/mirror/protocol.go` — Header struct
       - `services/provider-node/internal/mirror/protocol_test.go` — three new tests
       - `services/provider-node/internal/whip/whip.go:233` — VideoSPSPPS() origin capture
       - `services/provider-node/internal/whip/hls_muxer.go:67-78` — NewHLSMuxer signature
       - `.planning/phases/00-consolidation/00-PATTERNS.md` — evidence for Route A
       - ADR 0011 — Mirror-side HLSMuxer for origin-failure-tolerant HLS (predecessor)
       ```

|----------|-------------|
| origin WHIP SDP → origin Header.SPSPPSSet (via sess.VideoSPSPPS()) | SDP is attacker-controlled input from browser |
| origin Header → mirror ReadHeader | libp2p stream between trusted peer IDs |
| mirror Header.SPSPPSSet → HLSMuxer.spsCache/ppsCache | internal API, parameter sets trusted once in-process |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-04-01 | Tampering | Header.SPSPPSSet values | mitigate | Header body length capped at MaxFrameBytes=64KiB (protocol.go:77). Oversized SPS/PPS (>64KiB each) would already fail the frame read before JSON unmarshal. HLSMuxer receives parameter sets through the typed Go interface; mp4ff.avc.ParseSPSNALUnit (segmenter.go:290) validates before init-segment emission. |
| T-00-04-02 | Denial of Service | Malformed SPSPPSSet causes HLSMuxer panic | mitigate | Plan 00-05 MUST call `whip.NewHLSMuxer(sess.ID, sps, pps)` where a malformed pair triggers the existing error return (internal/whip/hls_muxer.go:78 err return). No panic-path introduced. |
| T-00-04-03 | Information Disclosure | SPS/PPS bytes visible on libp2p stream | accept | SPS/PPS are non-secret by design; they encode video dimensions + codec profile which are derivable from the RTP stream. |
