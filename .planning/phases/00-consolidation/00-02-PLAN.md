<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| provider-node process вҶ’ HTTP client | `/healthz` response is public (CORS allow-any); build hash leaks the short SHA of running binary |
| build-all.sh вҶ’ Go linker вҶ’ running binary | `-X main.Version=${VERSION}` writes the git short SHA into the binary at link time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-02-01 | Information Disclosure | /healthz .build field | accept | The short git SHA is already public information (merge commits visible on GitHub); exposing it on /healthz is equivalent to `GET https://github.com/Leeaandrob/aevia/commits/main`. No PII, no secrets. |
| T-00-02-02 | Tampering | main.Version at link time | accept | Attackers with access to rebuild the binary can already set arbitrary Version; that's a deploy-pipeline threat, not a /healthz one. The field is descriptive, not an authentication primitive. |
| T-00-02-03 | Information Disclosure | Version at compile time contains "dev" fallback | mitigate | build-all.sh:19 uses `${VERSION:-$(git rev-parse --short HEAD)}` which fails back to "dev" ONLY if git rev-parse returns empty AND the env var is unset; the rolling deploy in Plan 00-06 will reject a `build == "dev"` response as a deploy misconfiguration (explicit check in the deploy script). |
</threat_model>
